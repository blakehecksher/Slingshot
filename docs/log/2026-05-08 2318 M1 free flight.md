# 2026-05-08 2318 — M1 free flight

## What was done

- Split `main.ts` into modules per the planned layout:
  - `src/physics/world.ts` — Rapier WASM init + `PhysicsWorld` wrapper.
  - `src/render/scene.ts` — renderer, scene, lights, starfield, camera.
  - `src/game/input.ts` — gamepad + keyboard + (pointer-locked) mouse → unified `ShipCommand`.
  - `src/game/ship.ts` — Rapier dynamic rigid body + thrust + rotation.
  - `src/main.ts` — orchestrate, fixed-timestep loop, cockpit camera.
- Ship: dynamic body with a cuboid collider (1.5×0.8×2.5 m) sized for the placeholder hull. Density tuned so mass = 1 kg.
- Thrust applied as `applyImpulse(force × dt)` per physics tick — explicit one-shot impulses to avoid any ambiguity about Rapier's force-buffer clearing semantics.
- Rotation: direct `setAngvel` each tick from input, in ship-local axes rotated into world space. Letting go of stick → rotation stops snap.
- Cockpit camera mounted at body origin, inheriting ship orientation each render frame. Ship visual mesh hidden from cockpit view.
- HUD shows fps, dt, speed, gamepad-detected indicator, mouse-capture hint.
- Verified in headless Chrome via Playwright:
  - Init speed = 0 ✓
  - W (forward thrust) for 1s → ~30 m/s ✓ (matches FORWARD_THRUST = 30 m/s²)
  - Release thrust → speed unchanged across 2s drift ✓ (inertia preserved, no drag)
  - S (reverse) for 1s → ~13 m/s deceleration ✓ (matches REVERSE_THRUST = 18 m/s²)
  - 60fps sustained, console clean (favicon 404 only).

## What worked

- Splitting modules along the boundaries the plan suggested. They map well to the work.
- Using `applyImpulse(F * dt)` instead of `addForce`. Force-accumulator semantics across Rapier versions are too easy to get wrong; one-shot impulses just work.
- Direct angular-velocity control. Snappy, predictable, matches the arcade end of the spaceship-control spectrum. Will revisit if M2 reveals it doesn't pair well with gravity tug.

## What didn't and why

- **First pass with `setAdditionalMass(1.0)` and no collider was wrong.** Without a collider Rapier didn't derive a real mass, and forces produced absurd accelerations (~5 million× expected). Adding a cuboid collider with tuned density gives a real mass + inertia tensor, and the math behaved. Logging this as a footgun for future "why don't I need a collider yet" instincts.
- **First force application via `addForce` produced quadratic speed growth**, which suggests the force buffer wasn't being cleared between steps in this rapier-compat build (or my call site was stacking). Switching to `applyImpulse` with explicit per-tick `force × dt` fixed it cleanly. Decision: stay on `applyImpulse` — explicit, easy to reason about.

## Decisions made

- Use `applyImpulse(force × dt)` per tick rather than `addForce`. Reason: avoids the question of buffer-clearing semantics across Rapier versions; behavior is identical to a constant force when called every tick.
- Ship is a dynamic body with a cuboid collider whose collision groups are zeroed (M1 has nothing to collide with; M2+ can opt in deliberately).
- Direct angular velocity for rotation, not torque. Reason: snappier, easier to tune for "feel like flying" gate. Re-evaluate after M2.
